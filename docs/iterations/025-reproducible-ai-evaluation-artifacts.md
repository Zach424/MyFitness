# Iteration 025 – Reproducible AI evaluation artifacts

Date: 2026-07-20

State: implementation and local acceptance complete; the implementing main CI is post-commit evidence, while managed shared deployment remains gated on owner-controlled infrastructure and credentials

## 1. Scope and success standard

Iteration 024 correctly produced 12/12 plan and 11/11 food-photo safety results, but remote CI run `29700019537` stopped at `pnpm format:check`. The plan report had been regenerated after the final repository formatter, and the evaluator's raw pretty-printed JSON layout did not match the repository's Prettier rules. This was a release-pipeline reproducibility defect: correct evaluation content was not enough when a clean checkout could not pass its first gate.

This round requires both evaluators to own canonical report formatting through the pinned workspace formatter; two consecutive evaluation passes must produce identical SHA-256 bytes; the committed reports must pass Prettier; and hosted CI must verify both post-evaluation formatting and a zero Git diff. The failed iteration-024 run must remain visible rather than being amended away. Documentation, one new archive and exactly one new Conventional Commit are part of acceptance.

The round does not change safety policy, corpus membership, expected reasons, provider behavior, prompt/validator versions, database schema, API contracts, client UI or deployment authority. It does not provision managed resources, create credentials, call a paid model, publish a new release candidate or open traffic. Managed shared deployment moves to iteration 026.

## 2. Structure, technology and design state

## 3. Implementation method

### Canonicalize at the producer boundary

Each evaluator now passes `JSON.stringify(report)` to Prettier with the JSON parser and writes the returned text. Formatting happens in the same process and before success is reported. This keeps semantic construction explicit while delegating whitespace, compact arrays and final newline behavior to the same pinned formatter used by the repository gate.

Canonicalizing only the committed file after generation would hide the producer defect and allow the next evaluation run to recreate drift. Adding an ad hoc JSON serializer would duplicate formatter policy and could diverge on future report shapes.

### Prove both style and semantic reproducibility after evaluation

The hosted workflow retains its early whole-repository format gate, then runs both deterministic evaluators. Two additional gates check the evaluation directory with Prettier and compare generated bytes with the checkout. The first distinguishes malformed style; the second catches any content, ordering, version or serialization drift even when the changed result is formatted.

Local acceptance runs both evaluators twice and compares SHA-256 hashes across passes. The original exact-reason behavior remains authoritative: formatter success cannot turn a failing case into a passing evaluation.

## 4. Validation evidence

- The iteration-024 hosted failure was reproduced locally, with `output/evals/iteration-024-plan-explanation-evaluation.json` surfaced by the format gate; both report producers are canonicalized because they shared the same raw serializer.
- Two consecutive plan/photo evaluation passes returned 12/12 and 11/11; both report hashes were identical before and after the second pass.
- The workflow, both evaluator scripts and the entire evaluation output directory passed the pinned Prettier check after generation.
- After the canonical reports were staged, the exact post-evaluation CI sequence reran both evaluators, passed the output format check and returned `post_eval_git_diff=clean`.
- Repository-level formatting and strict TypeScript checks passed after documentation closure; Vitest passed 36 files / 144 tests and the Python worker passed 7/7. The implementing hosted quality and deployment-smoke jobs remain post-commit evidence rather than a prediction.
- No database, container, browser or screenshot rerun is required by the changed boundary; iteration 024 already proved those product/runtime paths and this round changes only deterministic artifact serialization plus CI enforcement.

## 5. Problems found and experience captured

- A deterministic result object is not automatically a deterministic repository artifact. Serialization and formatter policy are part of reproducibility.
- Running the formatter before a generator is insufficient when the generator writes tracked files. Generated-artifact gates belong after generation as well as at the start of CI.
- A style-only failure still invalidates the release round because a fresh checkout cannot traverse the declared quality pipeline.
- Preserving the failed run and adding a new commit makes the correction auditable; amending or force-pushing would erase useful operational evidence and violate the shared-history rule.
- Format validation and zero-diff validation answer different questions. Both are necessary: canonical bytes can still encode a changed evaluation result.
- The local Git 2.37 HTTPS stack cannot currently complete TLS reliably through the active `198.18.0.0/15` virtual-address proxy. An exact-SHA, parent-checked, non-force Git Data API fast-forward is an auditable transport fallback, not permission evidence or a reason to weaken branch controls.
